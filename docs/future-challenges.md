# OpenGENAI 開発ロードマップと今後の技術的課題

本ドキュメントでは、現在の OpenGENAI アーキテクチャ（マイクロサービス化・LiteLLMハブ構成）をベースとし、さらなる安定稼働、スケーラビリティ、および開発生産性の向上に向けた技術的課題と今後のアプローチを整理します。

---

## 1. 開発環境および本番環境でのリソース最適化（コンテナ管理）

### 課題
現在の構成では、主要サービス・推論エンジン・連携アプリを含め、全体で20の Docker コンテナ（サービス）が稼働しています。ローカル開発環境やリソースの限られたサーバでの検証時に、メモリやCPUの消費量が大きくなり、起動オーバーヘッドも増大しています。

### 今後のアプローチ
- **Docker Compose Profiles の徹底活用**:
  - 各マイクロサービスを機能ごとにプロファイル分類します（例: `audio` (Whisper系), `image` (Stable Diffusion系), `dify`, `rag`）。
  - 開発者がテキストチャットのコア機能のみを開発したい場合は、`docker compose up` で不要な推論系コンテナを起動させない運用をデフォルトとします。
- **軽量ローカルモデルのプロファイル化**:
  - `local-sd-api` や `ollama` に割り当てるモデルについて、CPU環境向けの極小モデル（例: `ollama pull gemma:2b` などの超軽量モデル）を選択可能にする設定を環境変数でトグル化し、開発マシンへの負荷を軽減します。

---

## 2. 共通モジュール（`shared`）の依存性整理と堅牢化

### 課題
`shared/` ディレクトリ配下にある共通コード（`docextract.py` や `ssrfguard.py`）は、複数の AIアプリ（exApps）やメインの `backend` にマウントまたはインポートされています。共通モジュールの変更が及ぼす影響範囲が広く、一部のサービスで予期せぬデグレードが発生するリスクがあります。

### 今後のアプローチ
- **共通モジュールの分離とパッケージ化**:
  - `shared` ディレクトリをローカルの Python パッケージ（例: `open-genai-shared`）として切り出し、`pyproject.toml` や `requirements.txt` でインストールする形式に移行します。
  - 各サービスが特定のバージョンを固定して利用できるようにすることで、急激な変更による破損を防ぎます。
- **CI/CD による影響範囲テストの自動化**:
  - 共通コードに変更が加えられた際、関連するすべての exApps の単体テスト（`pytest`）を自動的に実行・検証するマトリックスビルドを CI に追加します。

---

## 3. マイクロサービス間（backend ⇔ exApps）の API 整合性管理

### 課題
各 AIアプリ（exApps）は独立した FastAPI 等 of API サーバとして動作しています。`backend` と各 `exApp` 間の通信プロトコル（HTTP リクエスト/レスポンスの JSON スキーマ）の変更が、静的検証（静的型チェック）の段階で検知できず、実行時エラーになる懸念があります。

### 今後のアプローチ
- **スキーマ駆動開発 (Schema-Driven Development) の導入**:
  - API の型定義（Pydantic モデル等）を `shared` パッケージに集約し、`backend` と `exApps` の両方で同一のクラス定義をインポートして利用します。
  - または、ビルド時に各 exApp から自動生成した `openapi.json` を用いて、`backend` 側の通信クライアントコードがスキーマと一致しているかを静的にチェックするCIジョブを追加します。

---

## 4. 推論プロキシのガバナンスとキューイング機構

### 課題
ローカルで動作する画像生成（`local-sd-api`）や音声認識（`local-whisper-api`）は、GPU/CPU リソースを大量に消費するため、複数ユーザーから同時にリクエストが集中した際に OOM（Out Of Memory）の発生やレスポンスの極端な遅延（タイムアウト）が発生する可能性があります。

### 今後のアプローチ
- **非同期キューイング（メッセージブローカー）の導入**:
  - 画像生成や音声認識など、時間のかかる重いタスクについては、`backend` から直接 HTTP 同期呼び出しを行うのではなく、Redis や SeaweedFS のメタデータ通知を利用した non-blocking なタスクキュー（例: Celery や軽量なタスクランナー）を介した非同期処理へと移行します。
- **レート制限と同時実行制御（Toxicity & OOM Guard）**:
  - `sd-app` や `whisper-app` に同時実行数を制御するセマフォやレートリミッターを実装し、コンテナ側で安全にリクエストを受け流す（過負荷時には 429 Too Many Requests や待機ステータスを返す）ようにガードレールを構築します。
